# Wiki Schema — قوانین این پایگاه دانش

> این فایل را تو (ایجنت) خودت می‌خوانی و می‌نویسی. هرگاه عمیق‌تر فهمیدی،
> همین فایل را به‌روزرسانی کن. این تنها سند **بالا به پایین** wiki است.

## ساختار صفحات
ساختار سلسله‌مراتبی است — برای جستجوی سریع‌تر:

- `overview.md` — یک‌نگاه کلی پروژه (یک صفحه، ≤۴۰۰ کلمه)
- `index.md` — کاتالوگ همه صفحات با خلاصه یک‌خطی، گروه‌بندی شده
- `log.md` — تاریخچه append-only به ترتیب زمانی
- `entities/` — هر entity مهم یک صفحه (مدل دیتا، service، router)
- `concepts/` — patternها، flowها، قراردادها
- `sources/` — خلاصه فایل‌های خام کد که چندجا ارجاع داده می‌شوند

## لینک‌گذاری (Obsidian-style)
- از `[[entities/user]]` برای ارجاع به صفحه دیگر استفاده کن (نمونه‌ی نوشتاری)
- پسوند `.md` لازم نیست
- نام صفحات: lowercase + hyphen (مثال: `auth-flow.md` نه `Auth Flow.md`)
- **توجه:** wikilinkهای نمونه که داخل بک‌تیک یا code block باشند، در گراف لینک واقعی محسوب نمی‌شوند.

## چه وقت صفحه جدید بساز
- فقط اگر **≥۳ جای دیگر** قرار است به آن لینک دهد، یا یک concept مستقل با حجم قابل‌توجه است.
- در ۹۰٪ موارد، **به‌جای ساخت فایل جدید، صفحه موجود را آپدیت کن** (یک paragraph یا یک section اضافه).
- هرگز صفحه trivial نساز (مثلاً wrapper یک‌خطی، utility ساده). در صفحه پدر یک bullet بنویس.

## ساختار هر صفحه entity
```
# {نام}

> یک‌خطی توضیح هدف.

## مسئولیت‌ها
- ...

## وابستگی‌ها
- [[concepts/auth]] — چرا
- [[entities/database]] — چطور

## قراردادها / Edge cases
- ...

## منابع کد
- `path/to/file.py:42` — کلاس اصلی
```

## index.md سلسله‌مراتبی
```
# Index

## Overview
- [[overview]] — یک‌نگاه کلی

## Entities (X صفحه)
- [[entities/user]] — مدل کاربر، auth و subscription
- [[entities/project]] — ...

## Concepts (Y صفحه)
- [[concepts/auth-flow]] — flow کامل احراز هویت
- ...
```

## Lint (سلامت wiki)
وقتی کاربر گفت "wiki رو بازبینی کن":
1. orphan‌ها (بدون inbound link) — یا لینک بده یا حذف کن
2. broken link‌ها — صفحه‌ای که وجود ندارد را بساز یا لینک را حذف
3. تناقض بین صفحه‌ها — flag کن و بپرس کدوم درست است
4. صفحه‌های قدیمی (>۳۰ روز و کد مرتبط تغییر کرده) — بازخوانی

## وقتی session جدید شروع شد
1. **اول** `overview.md` و `index.md` را بخوان (هزینه: ≤۸۰۰ توکن).
2. اگر سوال خاصی است، صفحات مرتبط را drill down کن.
3. **هرگز** کل کدبیس را re-scan نکن.

## وقتی کاربر سوال جدیدی پرسید
- پاسخ را از wiki استخراج کن.
- اگر چیز ارزشمند کشف کردی، آن را در wiki ثبت کن (در صفحه مرتبط یا جدید).
- در `log.md` یک خط بنویس: `## [YYYY-MM-DD] query | <خلاصه>`.

## وقتی کد را تغییر دادی — قانون اکید
هر بار Write/Edit روی یک فایل پروژه (خارج از `.noqte/`):
1. تشخیص بده این تغییر روی کدام entity/concept در wiki اثر دارد.
2. صفحه‌ی مربوطه را به‌روز کن (فیلد جدید، حذف فیلد، تغییر امضای تابع، route جدید، dependency جدید).
3. اگر یک entity یا concept کاملاً جدید است، صفحه جدید بساز و در `index.md` لینک بگذار.
4. در `log.md` یک خط append کن: `## [YYYY-MM-DD] update | <خلاصه تغییر>`.

تغییرات trivial (typo، فرمت، rename متغیر بدون تغییر امضا) نیازی به آپدیت wiki ندارد. ولی هر تغییر **معنایی** باید منعکس شود — وگرنه wiki بی‌اعتبار می‌شود.
